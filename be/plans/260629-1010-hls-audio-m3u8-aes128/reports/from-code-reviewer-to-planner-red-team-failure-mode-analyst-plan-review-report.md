## Finding 1: Worker role boots full AppModule but never registers BullMQ — jobs queue forever, no consumer
- **Severity:** Critical
- **Location:** Phase 3, "Architecture" (conditional processor when `APP_ROLE === 'worker'`) and Plan "Risks" (role split)
- **Flaw:** The plan assumes the `worker` PM2 process will execute jobs. It will boot, but unless `HlsModule` is imported into `AppModule` AND `@nestjs/bullmq` `BullModule.forRootAsync` is registered in the standalone-context path, the worker boots an idle context. Worse: the plan gates only the *processor provider* on `APP_ROLE === 'worker'` (phase-03 line 19), but says nothing about gating the BullMQ *root connection* or ensuring the `worker` standalone context actually instantiates the `@Processor`. NestJS `@Processor` consumers only start when their module is instantiated in a running context — the plan never proves `HlsModule` is loaded in the `createApplicationContext` path.
- **Failure scenario:** API enqueues jobs (`status=pending` rows pile up). The `auth-be-worker` PM2 process is alive (`pm2 list` green) but its Nest context either didn't import HlsModule or the BullMQ connection wasn't created in worker mode. No job is ever consumed. `HlsAsset` rows sit `pending` indefinitely; `hlsUrl` never appears; no error is logged because nothing failed — it simply never ran. This is the single most likely silent failure.
- **Evidence:** `ecosystem.config.js:21-37` defines `auth-be-worker` with `APP_ROLE=worker`; `src/bootstrap.ts:144` routes non-api roles to `createApplicationContext(AppModule)` — a context, not an HTTP server; `src/common/app-role.util.ts:32-40` shows the ONLY role-conditional module wiring today is `buildScheduleImports` (scheduler only) — there is no worker-conditional module wiring precedent; `grep` for `bullmq|@nestjs/bullmq|Queue|Worker` across `src/` and `package.json` returned **zero** results — no queue infrastructure exists today. The plan must explicitly specify (a) where `BullModule.forRoot` registers, (b) that `HlsModule` is in `AppModule.imports`, and (c) that the processor instantiates under `createApplicationContext`.
- **Suggested fix:** Add an explicit Phase 3 step: register `BullModule.forRootAsync` in `AppModule` (or a role-gated import mirroring `buildScheduleImports`), confirm `HlsModule` is imported unconditionally, and add a smoke test that asserts a real job placed on Redis is drained by a `worker`-role context within N seconds. Do not rely on "worker role vẫn khởi tạo Nest context" (phase-03 line 27) as proof of consumption.

## Finding 2: Partial segment upload + crash leaves a `ready`-looking asset pointing at a broken playlist
- **Severity:** Critical
- **Location:** Phase 3, "Architecture" (`hls-r2.service.ts` "upload toàn bộ file dir") and "Risk Assessment" ("Nhiều PutObject")
- **Flaw:** The transcode produces 1 `.m3u8` + N `.ts` segments uploaded via N separate `PutObject` calls (no atomic multi-object commit on R2). The plan does not define upload ordering or a commit barrier. If the worker crashes / OOMs / is `pm2 restart`ed mid-upload (the worker has `max_memory_restart: "512M"` — ffmpeg + buffering large audio can hit that), some `.ts` are uploaded, some are not, and the playlist may already be up.
- **Failure scenario:** Worker uploads `index.m3u8` (or 30 of 50 segments), then crashes. On restart BullMQ retries the *same* job; but the partial objects already exist in R2. If the status was set to `ready` before all segments confirmed, the client fetches a playlist referencing `seg_031.ts`..`seg_050.ts` that 404 → playback stalls/dies mid-track. Even with retry, there's no cleanup of orphaned partial objects, and the unique constraint `@@unique([assetType, assetId])` (phase-02 line 34) means the retry overwrites in place with no versioning — a client mid-fetch can get a mix of old and new segments.
- **Evidence:** phase-03 line 24 "upload toàn bộ file dir lên `audio/hls/<assetType>/<assetId>/`" — no upload-order or atomicity spec; phase-03 line 25 processor "set `processing` khi bắt đầu" then later `ready` — no "upload all segments → verify → upload playlist last → then set ready" barrier; `ecosystem.config.js:28` `max_memory_restart: "512M"` makes mid-job restart realistic; `src/upload/audio-upload.service.ts:72-81` confirms uploads are single `PutObjectCommand` calls (no transactional batch). R2/S3 has no multi-object atomic commit.
- **Suggested fix:** Upload segments first, then upload the playlist **last** (playlist presence = commit marker), and only set `status=ready` after a HEAD/verify that every segment referenced by the playlist exists. Use a per-job content-addressed prefix (e.g. `.../<assetId>/<jobRunId>/`) so retries write a fresh dir and the playlistUrl flips atomically via the DB row, avoiding in-place overwrite races. Add cleanup of stale prefixes.

## Finding 3: Enqueue is fire-and-forget after a transaction that does NOT contain the HlsAsset write — silent data divergence
- **Severity:** Critical
- **Location:** Phase 5, "Requirements" / "Non-functional" ("fire-and-forget sau commit DB") and "Risk Assessment" (line 52)
- **Flaw:** Phase 5 says: upsert `HlsAsset(pending)` + generate/wrap key + `enqueueTranscode`, fire-and-forget after the entity commit, and "nếu enqueue lỗi không được rollback entity (log + để job backfill tay nếu cần)". But the entity create in `MusicService.create` is wrapped in `this.prisma.$transaction` (line 335-360). The plan never says whether the `HlsAsset` upsert joins that transaction or runs after. If the `HlsAsset` write runs **inside** the tx but enqueue runs after commit, a process crash between commit and enqueue leaves a `pending` HlsAsset with no job (orphan, needs manual backfill the plan explicitly defers). If the `HlsAsset` write runs **after** the tx, then the entity exists with no HlsAsset row at all on crash.
- **Failure scenario:** Admin creates 50 music tracks in a burst. For some, the API process restarts (deploy, OOM) between `$transaction` commit and the fire-and-forget enqueue. Those tracks have `audioUrl` set, possibly an orphan `pending` HlsAsset, but no queued job. No error surfaces to the admin (fire-and-forget). `hlsUrl` silently never appears for those tracks, and there is no reconciliation/sweeper — the plan defers backfill to "manual". Murphy guarantees this happens on the first production deploy that restarts the API mid-batch.
- **Evidence:** `src/music/music.service.ts:335-360` — entity created inside `$transaction`; phase-05 line 16 "enqueue không chặn response (fire-and-forget sau commit DB)"; phase-05 line 52 explicitly accepts orphaned `pending` with manual backfill. `src/chapters/chapters.service.ts:577-603` shows this codebase already uses `setImmediate(...).catch(() => silent)` fire-and-forget for unlock tracking — so a silently-swallowed enqueue failure is consistent with existing (risky) patterns and will not be noticed.
- **Suggested fix:** Make HlsAsset `pending` creation part of the same DB transaction as the entity write (transactional outbox row), and run a periodic reconciler (scheduler role already exists, `app-role.util.ts:28-30`) that finds `pending` HlsAssets with no active BullMQ job older than X minutes and re-enqueues. Do not rely on manual backfill. If enqueue must stay out-of-transaction, at minimum log at `error` (not silent) with the assetId so ops can react.

## Finding 4: Source mp3 download path from R2 is undefined — likely needs private GetObject that does not exist anywhere in the codebase
- **Severity:** High
- **Location:** Phase 3, "Risk Assessment" line 53 ("Tải nguồn mp3 từ R2 (private?) ... fallback GetObject")
- **Flaw:** The whole pipeline depends on the worker downloading the source mp3. The plan leaves "public GET vs private GetObject" unresolved as a risk to "check at Phase 3". But there is **no GetObject or presigned-URL code anywhere in the repo** — only `PutObjectCommand`. If the bucket is private (the norm for R2 with credentials), the worker must implement `GetObjectCommand` streaming from scratch, including handling that the stored URL (`r2AudioUrl`) is a `publicBaseUrl`-prefixed URL, not a key. There is no key-derivation-from-URL helper.
- **Failure scenario:** Plan is approved assuming "just download the URL." In Phase 3 the team discovers the bucket isn't publicly readable; a plain HTTP GET on `r2AudioUrl` returns 403. Every job fails `status=failed`. Or, if the bucket *is* public, then the source mp3 is world-readable — which undercuts the entire "AES-128 protects paid content" premise, since the unencrypted source is downloadable by anyone with the URL (and `r2AudioUrl` is deliberately hidden from API responses precisely because it's sensitive — see chapters service comments).
- **Evidence:** `grep GetObjectCommand|getSignedUrl|s3-request-presigner src/` returned **zero** results — no download capability exists; `src/upload/audio-upload.service.ts:81` builds public URLs as `${publicBaseUrl}/${key}`; `src/chapters/chapters.service.ts:46,83,203,226` repeatedly comment "audioUrl and r2AudioUrl intentionally omitted — use proxy" proving the raw R2 URLs are treated as not-for-public-exposure. The plan's premise (phase-02 line 21 of plan.md: ".ts public an toàn") collides with the unresolved source-access question.
- **Suggested fix:** Resolve bucket access policy *before* approving Phase 3 (it's a hard dependency, not a risk to defer). If private: specify `GetObjectCommand` streaming with timeout + bounded retry, and a helper to map stored URL → bucket key. If public: explicitly document that source mp3 is world-readable and confirm that's acceptable given the paid-content threat model (it likely is not).

## Finding 5: ffmpeg added to Dockerfile, but production deploys via deploy.sh on bare host — image and host install diverge
- **Severity:** High
- **Location:** Phase 3, "Architecture" / "Implementation Steps" step 5 ("Thêm ffmpeg vào Dockerfile runtime") and Plan "Risks" (Dockerfile thiếu ffmpeg)
- **Flaw:** The plan's only mitigation for the ffmpeg runtime dependency is "add ffmpeg to Dockerfile." But the actual production deployment path is `deploy.sh`, which SSHes to a bare host, untars source, runs `yarn build`, and `pm2 startOrReload ecosystem.config.js`. The Docker image is **not** used by this deploy flow. Adding ffmpeg to the Dockerfile does nothing for the PM2/SSH-deployed worker.
- **Failure scenario:** Team adds `ffmpeg` to `Dockerfile:55-57`, CI build goes green, plan acceptance criteria "Dockerfile runtime có ffmpeg" is checked off. Deploy via `deploy.sh` to host `72.62.198.196`. The `auth-be-worker` PM2 process spawns ffmpeg → `ENOENT: ffmpeg not found` on every job → all transcodes `status=failed`. The acceptance criterion was satisfied while the real deploy target was never provisioned.
- **Evidence:** `deploy.sh:195-329` is the production deploy — SSH + tar + `yarn build` + `pm2 startOrReload ecosystem.config.js`; it never builds or runs the Docker image. `ecosystem.config.js:22-37` runs the worker directly via `node ./dist/main.js` on the host. `Dockerfile:55-57` is where the plan wants ffmpeg, but that image is unused by `deploy.sh`. The plan does not mention provisioning ffmpeg on the host or adding a `command -v ffmpeg` preflight to `deploy.sh`/`ensure_yarn_toolchain`.
- **Suggested fix:** Add a host-provisioning step (install ffmpeg on the deploy host) and a preflight check in `deploy.sh` (`command -v ffmpeg && command -v ffprobe` for the worker) that fails the deploy loudly if missing. Also add a runtime guard in the processor that fails fast with a clear error if ffmpeg is absent. Keep the Dockerfile change too, but do not treat it as the production mitigation.

## Finding 6: BullMQ shares the cache Redis with no documented db/prefix isolation — eviction and FLUSH risk
- **Severity:** High
- **Location:** Phase 3, "Architecture" line 18 ("Connection dùng `cfg.redis.url`") and Plan "Dependencies" ("Redis dùng lại REDIS_URL hiện có")
- **Flaw:** BullMQ stores jobs, locks, and state as durable keys in Redis. The plan reuses the exact same `REDIS_URL` that backs `CacheModule` (cache-manager-redis-yet) with `ttl: 300`. BullMQ requires `maxRetriesPerRequest: null` on its connection and assumes Redis is **not** configured with `maxmemory-policy` eviction. A cache Redis is typically configured with an LRU/volatile eviction policy. If keys are evicted, in-flight job state/locks vanish → jobs lost or duplicated. There is also no `keyPrefix` separation defined.
- **Failure scenario:** Redis hits maxmemory under cache load (stories explore cache writes large objects — `stories.service.ts:418`). Eviction policy drops BullMQ keys (or a future `cache.reset()`/FLUSHDB intended to clear cache nukes the queue). Result: jobs silently dropped, locks lost causing the same transcode to run twice concurrently (double R2 upload, racing writes to the same `assetId` prefix → Finding 2 corruption), or `processing` jobs orphaned forever.
- **Evidence:** `src/app.module.ts:49-59` CacheModule uses `redisStore({ url: redisUrl })` with `ttl: 300` on the same `cfg.redis.url`; `src/shared/config/app-config.schema.ts:186-192` shows a single shared redis config (a `db` field exists but the cache factory doesn't pass it, so both would land on db 0 unless the URL specifies one); `stories.service.ts:418` writes sizable cache payloads making memory pressure plausible. No `keyPrefix` found in `src/`. BullMQ docs require non-evicting Redis and `maxRetriesPerRequest: null`.
- **Suggested fix:** Isolate the queue: use a dedicated Redis logical DB (or separate instance) for BullMQ, set an explicit `prefix`, and document that the queue Redis must have `maxmemory-policy noeviction`. Set `maxRetriesPerRequest: null` and `enableReadyCheck` per BullMQ requirements. Never point a FLUSH/`reset` at the shared db.

## Finding 7: "Extract pure entitlement function" underestimates side-effecting, divergent auth logic across chapter/variant/music
- **Severity:** High
- **Location:** Phase 4, "Architecture" (trích entitlement thuần) and "Implementation Steps" step 2 ("không đổi behavior")
- **Flaw:** Phase 4 assumes entitlement can be "extracted into a pure function (no I/O beyond Prisma)" shared between key endpoint and existing audio endpoints. But the real `getAudioUrl` entitlement is **not pure**: it fires `setImmediate` auto-unlock upserts as a side effect of *checking* access. The key endpoint will be called separately from the audio endpoint (hls.js fetches the key on its own request), so either the extracted function keeps the side effect (every key fetch writes an unlock row — double-counting, racing with the audio endpoint's own upsert) or it strips the side effect (behavior divergence between the two callers). The music path lives in a different service (`music-interaction.service.ts`) with its own playlist-unlock-for-track logic — "extract one shared pure function" across both is optimistic.
- **Failure scenario:** Key endpoint reuses the extracted check including the `setImmediate` upsert. hls.js requests the key once per playback (and HLS clients re-fetch keys). Now every playback writes/races `UserChapterUnlock` upserts from two endpoints concurrently → duplicate work, potential unique-constraint contention, skewed analytics. Alternatively the refactor strips the side effect and the *original* audio endpoint loses its unlock-tracking — a silent behavior regression the "không đổi behavior" criterion claims to prevent but won't catch (tests assert URL returned, not the fire-and-forget upsert).
- **Evidence:** `src/chapters/chapters.service.ts:577-587` and `594-603` — entitlement check performs `setImmediate` `userChapterUnlock.upsert` side effects (twice, for two branches); `getAudioUrl` signature (line 481-485) takes id/userId/variantId and mixes lookup + entitlement + side effect + URL resolution in one method — not cleanly separable; music access lives in `src/music/music-interaction.service.ts:233` (`findPlaylistUnlockForTrack`) — a different service with playlist-track-level access, so phase-04 line 19-22's "reuse" spans two distinct subsystems.
- **Suggested fix:** Define the extraction precisely: separate `checkEntitlement (pure read)` from `recordUnlock (side effect)`. The key endpoint must call only the pure read. Add a regression test asserting the audio endpoint still records the unlock and the key endpoint does **not**. Scope music separately rather than assuming one shared function.

## Finding 8: Music `playlist` type points audioUrl at another track's source — re-transcode and shared-asset ambiguity
- **Severity:** Medium
- **Location:** Phase 5, "Architecture" (Music create/update) and Plan "Open questions" (Music playlist)
- **Flaw:** For `contentType === playlist`, `audioUrl` is set to `resolvedPlaylistTracks[0]?.audioUrl` — i.e., the source URL of a *different* Music row (the first track). The plan keys `HlsAsset` on `(assetType='music', assetId=<playlist music id>)`, but the actual audio bytes belong to another music entity that likely already has (or will have) its own HlsAsset. This creates duplicate transcodes of identical source audio and ambiguous ownership. Worse, if the first track changes or is deleted, the playlist's HlsAsset silently points at stale/missing source.
- **Failure scenario:** Track A (music id A) and Playlist P (whose `audioUrl` = A's audioUrl) both get HlsAssets. A's audio is re-uploaded → A re-transcodes, but P's HlsAsset is never invalidated (the enqueue hook only fires on P's own create/update, and P's `audioUrl` field didn't change even though the underlying track did). P now serves an HLS playlist whose key/segments correspond to A's *old* audio. Or A is deleted → P transcodes a 404 source → `failed`.
- **Evidence:** `src/music/music.service.ts:306-313` — for playlist, `audioUrl = resolvedPlaylistTracks[0]?.audioUrl` (a borrowed URL, not P's own asset); phase-05 line 24-25 defers this to an open question and defaults to "track có audioUrl của music"; plan.md line 63 lists it as unresolved. The `@@unique([assetType, assetId])` (phase-02 line 34) cannot express "this music's audio is actually track A's audio."
- **Suggested fix:** Decide explicitly: do NOT create a separate HlsAsset for `playlist`-type music; instead have the playlist reference the first track's HlsAsset by track id, or skip HLS for playlist type entirely in v1 (it's already flagged out-of-scope-adjacent). Document that change-propagation across shared sources is unhandled and gate accordingly.

## Summary of Unresolved Questions for the Planner
1. Where exactly does `BullModule.forRoot` register, and is `HlsModule` imported in the standalone (`createApplicationContext`) path so the worker actually consumes? (Finding 1 — blocking)
2. Is the R2 bucket public or private, and what is the source-mp3 download mechanism? (Finding 4 — blocking; also has threat-model implications)
3. How does the production host (deploy.sh / PM2) get ffmpeg — not the unused Docker image? (Finding 5 — blocking)
4. Will the queue share the cache Redis db, and what is its eviction policy? (Finding 6)
5. Pure-vs-side-effecting entitlement: does the key endpoint record unlocks? (Finding 7)

## Verdict
The plan's local-validation (Phase 1) and crypto (Phase 2) phases are sound and low-risk. The async pipeline (Phases 3–5) rests on four unverified load-bearing assumptions — worker actually consumes jobs, R2 source is downloadable, ffmpeg exists on the real deploy target, and partial-upload/crash recovery — each of which is a silent-failure path confirmed against the current code. Recommend the planner resolve the five blocking questions and add an explicit reconciliation/atomic-commit story before implementation begins.
